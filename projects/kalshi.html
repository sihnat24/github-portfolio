Overview

What it is & why

Kalshi is a regulated prediction market exchange where participants trade contracts tied to real-world event outcomes, an unusual domain for automation because prices reflect crowd probability estimates rather than traditional asset dynamics. Prediction markets are smaller and less saturated than equity or crypto markets, making them a practical starting point for building algorithmic trading experience. This project is the data infrastructure layer: a persistent pipeline ingesting live market data with real-time visibility via Grafana, built as the foundation for a strategy layer currently in development.

Technical Approach

Architecture & decisions

The system is built around Python's asyncio. A WebSocket connection to Kalshi streams live orderbook deltas, which are queued and drained by a dedicated write worker, while REST polling runs concurrently for market snapshots and spot price data. This keeps ingestion and persistence decoupled so a slow write never blocks the live feed.

All data lands in InfluxDB, which we chose because it is a time series DB with simple Grafana integration. Three price sources are captured per asset: the Kalshi prediction market mid-price, the CoinGecko spot price, and the CF Benchmarks RTI settlement price. The last of these is the authoritative reference Kalshi uses for contract resolution, making it the critical anchor for any future strategy work. Orderbook deltas are stored with sequence numbers and write timestamps, enabling latency measurement between exchange event and database write.

The entire stack runs in Docker Compose and is deployed to a VPS for continuous 24/7 operation. The bot automatically rolls to the next active contract as each 15-minute market expires, requiring no manual intervention between sessions.

Results

Outcomes & metrics

The pipeline ingests approximately 29 million orderbook deltas every six hours, roughly 1,350 writes per second and 1.2 million events per 15-minute contract window. Each window produces two streams, as Yes and No contracts are tracked independently. Figure 1 shows the live delta ingestion rate as visualized in Grafana, and Figure 2 shows the concurrent BTC market data feed sourced from CoinGecko's REST API, which returns a JSON payload covering price, volume, and broader market context. The system handles WebSocket disconnects and contract rollovers automatically, requiring no manual intervention. A heartbeat mechanism monitors the ingestion process continuously and restarts it if it goes down, ensuring the pipeline stays live around the clock.

Planned extensions include replacing the CoinGecko REST polling layer with a direct Kalshi WebSocket feed for real-time BTC price and candlestick data, bringing spot price ingestion in line with the low-latency orderbook pipeline. From there the focus shifts to building out a backtesting framework against the stored time-series data, followed by a live trading layer that can act on the signals the data exposes.
